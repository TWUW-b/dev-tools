テストケース MD の書き方
========================

このライブラリのテストケースは必ず **Domain → Capability → Case** の3階層。
テスターは「自分はこのロールだ」と決めて1つの domain ファイルを上から順に試すので、
**1 ファイル = 1 ユーザーロール**（＝1 domain）でまとめること。機能軸で分けない。

- frontmatter `domain`: ユーザーロール名（例: guest / user / client / admin / app-admin）
- H1 `#`: そのロールが触る機能グループ（capability）
- リスト `-`: 実際に試す検証項目（case）

NG 例: 認証・ユーザー管理など機能軸で MD を分けて複数ロールを混在させる
OK 例: user.md / admin.md のようにロールごとにファイルを作り、中に機能を並べる
-->

# ログイン・アカウント
- メールアドレスとパスワードでログインできる
- パスワードを忘れた場合にリセットメールを受け取れる
- ログアウトできる
- プロフィール画像を変更できる

# ダッシュボード
- ログイン直後に自分のダッシュボードが表示される
- 未読通知の件数が表示される

# データ登録
- 新規データを登録できる
- 必須項目未入力でエラーメッセージが表示される
- 登録後に一覧へ反映される

# データ編集・削除
- 自分が作成したデータのみ編集できる
- 削除時に確認ダイアログが出る
