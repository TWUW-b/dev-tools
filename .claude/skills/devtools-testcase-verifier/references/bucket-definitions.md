# バケット定義と判定ルール

テストケース検証の結果は **5 バケット** のいずれかに必ず振り分けます。

## 5 バケット

| バケット | 条件 | 対応 |
|---|---|---|
| ✅ **OK** | 手順通り実行でき期待結果と一致 | 記録のみ |
| 🔧 **TC_WRONG** | 実装は意図通りだが TC 側の記述・期待・手順が食い違い（OBSOLETE 含む） | 記録のみ |
| 🐛 **IMPL_BUG** | TC の期待が正しい仕様で、実装がそれを満たしていない | 記録のみ（修正は別会） |
| ❓ **OTHER** | 判定保留（環境要因 / データ不足 / 情報不足 / 一時的要因 / MCP 制約） | 理由 + 次アクション記録 |
| ⏸ **SKIP** | 破壊的操作 / 承認未取得 / 明示的スキップ指示 | スキップ理由を記載 |

## 判定フローチャート

```
検証開始
  ↓
手順通りに実行できたか？
  ├─ No → ❓ OTHER（理由: 実行不可）
  └─ Yes
       ↓
  期待結果と一致したか？
    ├─ Yes → ✅ OK
    └─ No
         ↓
    TC の期待が実装の意図と合っているか？
      ├─ No（TC 側が古い/間違い） → 🔧 TC_WRONG
      └─ Yes（実装が期待を満たさない）
           ↓
      MCP 固有の症状の可能性はあるか？
        ├─ Yes → ❓ OTHER（人間操作で再現確認待ち）
        └─ No → 🐛 IMPL_BUG
```

## 判定ルール

### R1. evidence 必須

**evidence なしの判定は原則不可**。最低 1 件の screenshot を evidence/ に保存する。

- ✅ OK: 期待結果が画面に現れた時点で screenshot
- 🔧 TC_WRONG: 実装側の現状を screenshot（TC との差分が分かる状態で）
- 🐛 IMPL_BUG: 再現 screenshot + 複数アングル推奨
- ❓ OTHER: 失敗状態または判定保留に至った画面
- ⏸ SKIP: evidence 不要（スキップ理由のみ記載）

### R2. MCP 固有症状の扱い

**MCP 自動操作固有の症状を IMPL_BUG と断定しない**。

以下は MCP 制約の可能性が高く、人間操作で再現確認するまでは `❓ OTHER` 扱い:

- 要素が存在するのに `click` が反応しない
- `wait_for` がタイムアウトする
- カスタム UI コンポーネント（日付ピッカー、モーダル）が操作不能
- iframe 内要素の操作が失敗
- フォーカス依存の操作（keyboard event 等）

実装バグと確定するには **人間による再現** が必要。その旨を OTHER の `note` に明記する。

### R3. 操作後のフォロースルー

操作がタイムアウトした場合、まず以下を確認してからリトライ:

1. ブラウザダイアログ（alert / confirm / prompt）が出ていないか
2. コンソールにエラーが出ていないか（`list_console_messages`）
3. ネットワーク層で 5xx / 認証切れが発生していないか

原因を切り分けずにリトライを重ねない。詳細は `chrome-mcp-guidelines.md` G10 を参照。

### R4. TC_WRONG vs IMPL_BUG の見分け方

ケースによってはどちらか判断が難しい。以下の基準で切り分ける:

| 観点 | TC_WRONG | IMPL_BUG |
|---|---|---|
| 実装の挙動が、仕様や他画面と一貫している | ✅ | — |
| 実装の挙動が、ユーザー体験として明らかに壊れている | — | ✅ |
| TC の表現が旧機能名 / 旧 URL を参照している | ✅ | — |
| 実装が直近のリファクタで機能を移動した | ✅ | — |
| 同種の操作が他画面で正常動作する | — | ✅ |
| セキュリティ / データ破壊の可能性がある | — | ✅ |

判断がつかない場合は `❓ OTHER` で保留し、別会で再評価する。

### R5. OTHER の下位分類

OTHER が多すぎる round では、次回の解消戦略を立てるため **理由の分類** を note に明記する:

| 理由タグ | 典型 |
|---|---|
| `[DATA-MISSING]` | テストデータ未投入（ファイル、公開リンク等） |
| `[MCP-LIMIT]` | Chrome MCP の制約（v-calendar、カスタムドロップダウン等） |
| `[AUTH-EXPIRED]` | セッション・トークン期限切れで再現不可 |
| `[QUOTA]` | 上限系（大量データ投入が必要） |
| `[TIMING]` | 時間経過が必要（7 日待機等） |
| `[PERMISSION]` | 管理者権限・admin-key 不足 |
| `[UNCLEAR]` | TC の記述だけでは判定基準が不明 |

note 例: `❓ OTHER [MCP-LIMIT] v-calendar の日付ピッカーが操作できず、計測日入力不可`

### R6. SKIP の条件

以下は **明示的にスキップして良い** ケース:

1. 破壊的操作（本番データ削除、課金発生、外部通知送信等）
2. 承認未取得の環境（本番 DB、共有アカウント等）
3. 既存データを変更する操作（検証用データに閉じれない）
4. ユーザーが明示的に「今回はやらない」と指示した

SKIP 時は必ず **理由** を記載する。

## 進捗サマリ表の書式

`01_checklist.md` 冒頭に以下の形式で集計:

```markdown
| Phase | 対象 | 全 | 未実施 | ✅ OK | 🔧 TC_WRONG | 🐛 IMPL_BUG | ❓ OTHER | ⏸ SKIP |
|---|---|---|---|---|---|---|---|---|
| 7 | CA セットアップ | 31 | 0 | 20 | 5 | 0 | 4 | 2 |
| 8 | CA ファイル | 16 | 0 | 10 | 1 | 0 | 5 | 0 |
| | **合計** | **47** | **0** | **30** | **6** | **0** | **9** | **2** |
```

`update-checklist.mjs` が自動で再計算するので手動編集不要。
