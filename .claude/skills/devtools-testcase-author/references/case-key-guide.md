# case_key とアンチパターンの詳細ガイド

本ドキュメントは `devtools-testcase-author` skill の補助リファレンスです。
テストケース MD を書く人・レビューする人が必ず参照してください。

---

## 1. case_key の仕組み

`@twuw-b/dev-tools` v1.2.0 以降、`test_cases` テーブルは `case_key` という **不変 ID** を一意キーとします。タイトルや capability を書き換えても、同じ `case_key` であれば既存行を UPDATE し、`test_runs` (実行履歴) と `notes` (バグ報告紐付け) は維持されます。

### フォーマット

```
TC-{role_code}-{連番}
```

- `role_code`: 2〜6 文字の英大文字 + 数字。プロジェクトごとに決める
- 連番: そのファイル内の `TC-{role_code}-` 最大値 + 1。欠番 OK、再利用禁止
- 3 桁 → 4 桁への桁上げは自動（既存 `001` を `0001` に書き換える必要はない）

### 例

```
TC-AA-001   ← アプリ管理者ロール、最初のケース
TC-AA-042   ← 同ロール、42 番目
TC-CA-105   ← 企業管理者ロール、105 番目
TC-AC-007   ← 認可・アクセス制御ドメイン、7 番目
```

### 不変性の絶対ルール

1. **一度採番したら二度と番号を変更しない**
2. **欠番を埋めるために若い番号を再利用しない**
3. タイトル・domain・capability は自由に書き換え可。`case_key` が同じであれば UPSERT される
4. ケースを廃止したい場合は MD から行ごと削除する（`case_key` を残しつつコメントアウトしない）

## 2. role_code の決め方

プロジェクトごとに 2〜4 文字の英大文字コードを決めます。同一プロジェクト内でユニークで、ロール名から連想できる文字列を選びます。

### skill デフォルト (英語ロール名ベース)

| ロール | role_code | ファイル |
|---|---|---|
| guest | `GU` | guest.md |
| user | `US` | user.md |
| client | `CL` | client.md |
| admin | `AD` | admin.md |
| app-admin | `AP` | app-admin.md |
| access-control | `AC` | access-control.md |

### プロジェクト固有の例（タスク管理 SaaS）

| ロール | role_code | ファイル |
|---|---|---|
| システム管理者 | `SA` | 01-sys-admin.md |
| チームマネージャー | `TM` | 02-team-manager.md |
| メンバー | `MB` | 03-member.md |
| 閲覧ユーザー | `VW` | 04-viewer.md |
| ゲスト | `GU` | 05-guest.md |
| 認可・アクセス制御 | `AC` | 06-access-control.md |

### frontmatter 必須

```yaml
---
domain: 企業管理者
role_code: CA
---
```

- `domain`: 表示用の日本語名（DebugPanel 上のグルーピングラベル）
- `role_code`: 2〜6 文字の英大文字+数字。**そのファイル内の全 case_key の接頭辞と一致必須**

## 3. アンチパターン A: UI で検証できないテストを書かない

### 前提

テスターは `@twuw-b/dev-tools` の `DebugPanel` の TestFlow UI からチェックボックスで pass/fail を記録します。手元には以下がありません:

- データベース管理ツール (pgAdmin / DBeaver 等)
- API コンソール (Postman / curl)
- サーバーログ
- 内部 ID / タイムスタンプの確認手段

したがって **ブラウザの画面操作だけ** で pass/fail を判定できるケースしか書いてはいけません。

### NG 例と書き換え

| ❌ NG (UI 確認不能) | ✅ OK (UI 観察可能) |
|---|---|
| 招待受諾: 受諾済招待の役割が DB に正しく反映される | 招待受諾: 受諾後、対象ユーザーが該当企業のメンバー一覧に表示される |
| ユーザー登録: users テーブルに新規行が INSERT される | ユーザー登録: 登録完了画面に入力したメールアドレスが表示される |
| API レスポンスの JSON スキーマが仕様と一致する | 画面の該当エリアに取得データが表示される |
| createdAt が UTC で保存される | 作成日時が画面にローカルタイムで表示される |
| ログ出力に特定のメッセージが記録される | エラー通知トーストがユーザー画面に表示される |
| パスワードが bcrypt でハッシュ化されている | ログイン画面で正しいパスワードを入れると成功し、誤ったパスワードだと失敗する |

### 判定基準

ケースを書く前・書いた後に自問する:

> **「テスターが DebugPanel の TestFlow UI からこのケースを実行したとき、ブラウザ画面のどこに結果が現れるか？」**

- 「○○画面の△△が表示される」「ボタンを押すと××に遷移する」等、**画面上の具体的な変化** を言語化できればOK
- 「DB に反映される」「API が呼ばれる」等、裏側の挙動しか言えないなら **書き換えるかアンチパターン A に該当** として却下

### 補足: DB 検証が必要な場合

内部状態の検証は **ユニットテスト or 結合テスト** の領分です。`docs/test-cases/` の MD はあくまで **QA テスター向けの受入テスト** であり、両者を混ぜないでください。

- ユニットテスト: `src/**/*.test.ts`
- 結合テスト: `tests/api/*.test.ts`
- 受入テスト: `docs/test-cases/*.md` ← **本 skill の対象**

## 4. アンチパターン B: 認可境界テストをロール MD に混ぜない

### 問題

ロール別 MD (例: `02-team-manager.md`) はそのロールの **日常業務フロー** を上から順に実行する想定です。テスターは「チームマネージャー」としてログインし、業務メニューを順に試します。

ここに認可境界テスト (= 他ロール・他テナントのデータに触れないことの確認) を混ぜると:

1. **不自然な動線**: 業務を試している最中に「急に他企業の URL を直打ちしてみる」不連続な動作が必要
2. **複数アカウント切替**: 認可テストには「A 企業管理者」「B 企業管理者」の 2 アカウントが必要で、1 ロール視点の MD に収まらない
3. **失敗時の重大度の違い**: データ漏洩は通常機能の UI バグとは優先度が段違い。混ざっていると issue トリアージ時に見落とされる

### 解決策: 認可ドメインを別ファイルに分離

`docs/test-cases/06-access-control.md` (skill デフォルト) を新設し、そこに全ロールの認可境界テストを集約します。

```md
---
domain: 認可・アクセス制御
role_code: AC
---

# 01 企業・テナント境界
- [TC-AC-001] 企業管理者: 他企業の管理画面 URL を直打ちしても該当画面が開かず、エラーかログイン画面にリダイレクトされる
- [TC-AC-002] 企業管理者: 他企業のユーザー編集 URL を直打ちしても編集フォームが表示されない
- [TC-AC-003] 一般ユーザー: 共有されていない他企業のファイル詳細 URL に直接アクセスしても画面が開かない

# 02 ゲスト認可
- [TC-AC-101] ゲスト: 保護された画面 (/dashboard 等) に直接アクセスするとログイン画面に遷移する
```

### 認可ドメインの書き方ルール

1. **各ケースの冒頭にテスト主体ロールを書く**: `企業管理者:` `一般ユーザー:` `ゲスト:` 等。テスターがどのアカウントで実行するか一目で分かるように
2. **アンチパターン A と同様、UI 観察可能なケースに限定**: `API レベルでもアクセスできない` → `直打ちしても画面が開かない` に書き換え
3. **capability は権限軸で切る**: `企業・テナント境界` / `ゲスト認可` / `読み取り専用権限` / `権限昇格の拒否` / `セッション・認証`
4. **優先度ラベル**: 認可系は原則 critical/high なので、DebugPanel 上で optional にマークされる運用にする (プロジェクト側の仕組みに応じて)

### 該当ロール MD から抽出すべきもの

既存のロール MD に以下のパターンがあれば access-control.md に移動:

- `他企業のデータにアクセスできない`
- `権限のないユーザーはアクセス拒否される`
- `URL 直打ちでリダイレクトされる`
- `認可エラーが表示される`
- `編集ボタンが表示されない` (権限軸の場合)
- `ログアウト後に保護画面が見えない`

## 5. よくあるエラーと対処

| エラー | 原因 | 対処 |
|---|---|---|
| `missing frontmatter` | `---` ブロックが無い | ファイル先頭に `---\ndomain: ...\nrole_code: ...\n---` を追加 |
| `domain required in frontmatter` | `domain:` 行が無い | frontmatter に追加 |
| `role_code required in frontmatter` | `role_code:` 行が無い | 同上 |
| `missing [TC-XX-NNN] key` | `- ` の直後にキー記法がない | `- [TC-XX-NNN] タイトル` 形式に |
| `case_key prefix mismatch` | frontmatter `role_code` と case_key の接頭辞不一致 | どちらかを揃える |
| `duplicate case_key in file` | 同一ファイル内で case_key 重複 | 片方を別番号に |
| `duplicate case_key across files` | ファイル横断で重複 | 別番号に修正 |
| `case before any H1 capability` | `# 見出し` より前に `- ` がある | `# 見出し` を先に書く |

## 6. 参考: dev-tools v1.2.0+ の UPSERT 動作

`POST /__debug/api/test-cases/import` は v11 以降 **case_key ベースの UPSERT**:

1. `case_key` で既存行を検索
2. 見つかれば domain/capability/title を UPDATE (archived_at も NULL に戻す)
3. 見つからず、同じ domain/capability/title の `LEGACY-*` 行が archived で存在すれば、その行の case_key を書き換えて再利用 (v11 バックフィル対応)
4. それ以外は INSERT

したがって **タイトル変更 = 新規行生成ではなく既存行の更新**。`test_runs` と `notes` は一切変更されません。

### 削除したい場合

本 skill の `import-test-cases.mjs` は **一切削除操作を行いません**。廃止したいケースは:

1. MD から該当行を削除して import
2. 本スクリプトが `[warning] MD から消えた case_key: N 件` を出力
3. 利用者が手動で dev-tools 管理 UI の archive 機能を使うか、`DELETE /__debug/api/test-cases` を手動呼び出し (ソフトデリートなので履歴は残る)
4. 物理削除はサポート外

## 7. チェックリスト (PR レビュー用)

テストケース MD を追加・編集する PR を作る前に確認:

- [ ] frontmatter に `domain:` と `role_code:` がある
- [ ] すべての `- ` 行に `[TC-XX-NNN]` キーが付いている
- [ ] キーの接頭辞が frontmatter の `role_code` と一致する
- [ ] 既存キーの番号を変更していない
- [ ] 新規追加は「そのファイル内の最大値 + 1」で採番
- [ ] 認可境界テストを該当ロール MD に書いていない (access-control.md に書いている)
- [ ] すべてのケースが **ブラウザ画面上で観察可能** (DB 検証・API レベル検証が含まれない)
- [ ] `node scripts/import-test-cases.mjs http://localhost:8082/api/__debug --dry-run` が通る
